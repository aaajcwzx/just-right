# 贡献指南

感谢您对「刚刚好」插件的关注！我们欢迎任何形式的贡献。

## 🤝 如何贡献

### 报告 Bug

在提交 Bug 前，请先搜索现有的 [Issues](../../issues) 确认问题是否已被报告。

提交 Bug 时请包含：
- 浏览器版本（Chrome/Edge）
- 插件版本
- 复现步骤
- 预期行为 vs 实际行为
- 截图（如果适用）

### 功能建议

我们欢迎新功能建议！请通过 Issue 描述：
- 功能的使用场景
- 预期的用户体验
- 可能的实现方式（可选）

### 提交代码

1. **Fork 本仓库**

2. **创建功能分支**
   ```bash
   git checkout -b feature/your-feature-name
   ```

3. **本地开发**
   - 修改代码
   - 在浏览器中测试（chrome://extensions/）
   - 确保符合代码规范

4. **提交变更**
   ```bash
   git add .
   git commit -m "feat: 添加新功能描述"
   ```

5. **推送到远程**
   ```bash
   git push origin feature/your-feature-name
   ```

6. **创建 Pull Request**
   - 描述清楚改动内容
   - 关联相关 Issue
   - 添加截图（如果是 UI 改动）

## 📝 代码规范

### JavaScript 风格

- 使用 ES6+ 语法
- 使用 2 空格缩进
- 使用 `const` 和 `let`，避免 `var`
- 异步操作使用 `async/await`
- 函数名使用驼峰命名
- 常量使用大写下划线命名

### 文件组织

```
刚刚好/
├── *.html          # 页面文件
├── *.js            # 脚本文件（一个页面对应一个脚本）
├── *.css           # 样式文件
├── manifest.json   # 插件配置
├── background.js   # 后台服务
└── icons/          # 图标资源
```

### 提交信息规范

使用约定式提交格式：

- `feat:` 新功能
- `fix:` Bug 修复
- `docs:` 文档更新
- `style:` 代码格式（不影响功能）
- `refactor:` 重构
- `perf:` 性能优化
- `test:` 测试相关
- `chore:` 构建/工具相关

示例：
```
feat: 添加暗黑模式支持
fix: 修复通知不显示的问题
docs: 更新安装说明
```

## 🔒 安全注意事项

- 不要在代码中硬编码敏感信息
- 验证所有用户输入
- 遵循 CSP 策略（无内联脚本/样式）
- 数据导入时进行类型和范围检查
- 保护用户隐私，通知文案不泄露敏感信息

## 🎨 UI/UX 设计原则

参考 PRODUCT.md 中的设计理念：

1. **克制** - 工具不应该喧宾夺主
2. **专业** - 可信赖的视觉语言
3. **低调** - 隐私友好的设计

设计风格：
- 使用 OKLCH 颜色系统
- 避免过度的圆角和阴影
- 避免彩色渐变
- 保持简洁的排版

## 🧪 测试

在提交 PR 前，请测试：

- ✓ 安装/卸载插件
- ✓ 基本提醒功能
- ✓ 练习流程（开始、暂停、跳过、完成）
- ✓ 设置页面所有选项
- ✓ 数据导入/导出
- ✓ 键盘快捷键
- ✓ 隐私模式
- ✓ 成就解锁
- ✓ 在 Chrome 和 Edge 中测试
